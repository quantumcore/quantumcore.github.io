The source code revealed that the page content was not directly written in HTML. Instead, it was injected dynamically using JavaScript obfuscation:

- The main payload was encoded using percent-encoding (`%3C%21%44%4F...`)
- It was decoded at runtime using `unescape()` and rendered via `document.write()`

A common technique to bypass static analysis.

The most notable aspect of this sample was the deliberate use of **multi layer client-side obfuscation** to conceal both structure and intent. While the underlying attack logic was relatively straightforward. 
### Encoded Payload via Percent-Encoding
The outer HTML document did not directly contain executable logic. Instead, it relied on a single JavaScript call:
```js
document.write(unescape('%3C%21%44%4F%43%54%59%50%45...'));
```

This represents a **percent-encoded payload**, where every byte of the original HTML/JavaScript content was transformed into hexadecimal escape sequences prefixed with `%`.

For example:

- `%3C` → `<`
- `%68` → `h`
- `%74` → `t`
- `%6D` → `m`
- `%6C` → `l`

When combined, these sequences reconstruct a fully functional HTML document.
### Runtime Reconstruction Using `unescape()`

The decoding process occurs entirely at runtime through the deprecated but still functional `unescape()` function.

In this case, I observed the following execution chain:
1. The browser loads the initial HTML shell
2. The JavaScript engine encounters `document.write()`
3. `unescape()` decodes the percent-encoded string into raw HTML
4. The decoded content is injected directly into the DOM
At this point, the real phishing interface is never visible in the static source view, it only exists **after execution in the browser context**.

This is a key evasion technique because:
- View-source inspection shows only obfuscated data.
- Security tools that do not execute JS miss the payload entirely.

### Honeypot Field

A hidden input field was included:

```html
<input type="text" name="_gotcha" style="display:none">
```

 A simple bot detection method. Automated scrapers often fill all fields, while real users never interact with hidden inputs. Any value in this field can be used to identify non-human interaction.
 
## Credential Flow and Exfiltration Path

After reconstructing the full execution flow, the most important finding was not the UI deception itself, but the **data exfiltration** that activates immediately after user input.
The browser never sends credentials to a legitimate authentication backend. Instead, the form submission is intercepted entirely on the client side:

- Email and password are captured via DOM access
- Input values are normalized using `.trim()`
- Data is assembled into a structured JSON object
